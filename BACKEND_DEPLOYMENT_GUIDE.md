## Quick Start (When PHP Available)

```bash
# 1. Navigate to server directory
cd c:\Sheltra\sheltra_client_side\server

# 2. Install dependencies (first time only)
composer install

# 3. Create .env file from example
copy .env.example .env

# 4. Generate application key
php artisan key:generate

# 5. Run migrations (when DB is ready)
php artisan migrate

# 6. Start development server
php artisan serve
```

API will be available at: `http://localhost:8000/api`

## Architecture Overview

### Technology Stack
- **Framework**: Laravel 10.x (actually 8.75+ from composer.json)
- **Authentication**: Session-based (web) + Sanctum tokens (API)
- **Authorization**: Role-based access control (RBAC)
- **Database**: MySQL/PostgreSQL (configurable via .env)
- **Frontend**: React/Vite on localhost:3000

### Core Roles
```
- refugee: Skill verification seekers
- ngo: Partner organizations for verification
- employer: Organizations hiring refugees
- admin: Platform administrators
```

## API Endpoints Structure

### Authentication Routes (Public)
```
POST   /api/auth/register         - User registration
POST   /api/auth/login            - User login
POST   /api/auth/logout           - User logout (requires auth)
POST   /api/auth/forgot-password  - Password reset request
POST   /api/auth/reset-password   - Password reset submission
```

### Auth State Routes (Public)
```
GET    /api/auth/me               - Get current user info
POST   /api/auth/validate         - Validate session/token
```

### Refugee Routes (role:refugee)
```
GET    /api/refugee/profile       - Get profile
POST   /api/refugee/profile       - Create/update profile
PUT    /api/refugee/profile       - Update profile
GET    /api/refugee/opportunities - Get matched opportunities
GET    /api/refugee/verification-status - Check skill verifications
POST   /api/refugee/skills        - Update skills
GET    /api/refugee/applications  - Get job applications
```

### Employer Routes (role:employer)
```
GET    /api/employer/profile      - Get company profile
POST   /api/employer/profile      - Create/update profile
POST   /api/employer/jobs         - Create job posting
GET    /api/employer/jobs         - List job postings
PUT    /api/employer/jobs/{id}    - Update job posting
GET    /api/employer/applicants   - List applicants
POST   /api/employer/feedback/{refugeeId} - Submit feedback
```

### NGO Routes (role:ngo)
```
GET    /api/ngo/cases             - Get assigned cases
GET    /api/ngo/cases/{caseId}    - Get case details
POST   /api/ngo/cases/{caseId}/verify/{refugeeId} - Submit verification
POST   /api/ngo/cases/{caseId}/notes - Add case note
GET    /api/ngo/cases/{caseId}/notes - Get case notes
GET    /api/ngo/metrics           - Get NGO metrics
```

### Admin Routes (role:admin, requires check.admin middleware)
```
GET    /api/admin/metrics         - Get platform metrics
GET    /api/admin/users           - List all users
GET    /api/admin/ngos            - List NGO partners
GET    /api/admin/audit-logs      - Get audit logs
POST   /api/admin/users/{userId}/suspend - Suspend user
POST   /api/admin/users/{userId}/reactivate - Reactivate user
```

## API Response Format

All endpoints follow a consistent JSON response structure:

### Success Response
```json
{
  "success": true,
  "message": "Operation successful.",
  "data": {
    // Endpoint-specific data
  }
}
```

### Error Response
```json
{
  "success": false,
  "message": "Error description",
  "errors": {
    "field_name": ["Validation error message"]
  }
}
```

### Validation Error Response (422)
```json
{
  "success": false,
  "message": "Validation failed",
  "errors": {
    "email": ["Email already registered"],
    "password": ["Password must be at least 8 characters"]
  }
}
```

## Authentication Flow

### Session-Based (Web/SPA)
```
1. POST /api/auth/register or /api/auth/login
2. Server creates session, sets HTTP-only cookie
3. Subsequent requests include cookie automatically
4. SessionController.currentUser() returns authenticated user
5. POST /api/auth/logout destroys session
```

### Sanctum Tokens (Future)
```
1. POST /api/auth/login returns { token: "..." }
2. Client stores token locally
3. Send Authorization: Bearer {token} in headers
4. Server validates token via Sanctum middleware
```

## Testing Checklist

### 1. Bootstrap Test
```bash
php artisan list
```
✓ Should show Laravel artisan commands

### 2. Configuration Test
```bash
php artisan config:show app
```
✓ Should show APP_NAME as "Sheltra"

### 3. Authentication Test
Register and login a test user:
```bash
curl -X POST http://localhost:8000/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Test User",
    "email": "test@example.com",
    "password": "password",
    "password_confirmation": "password",
    "role": "refugee"
  }'
```

### 4. Protected Route Test
```bash
curl -X GET http://localhost:8000/api/refugee/profile \
  -H "Cookie: XSRF-TOKEN=...; laravel_session=..."
```
✓ Should return 200 with profile data or 401 if not authenticated

### 5. Role Check Test
Try accessing admin endpoint with refugee user:
```bash
curl -X GET http://localhost:8000/api/admin/metrics \
  -H "Cookie: laravel_session=..."
```
✓ Should return 403 "Admin access required"

## Troubleshooting

### "Could not open input file: artisan"
- Verify you're in the `server/` directory
- Check PHP is installed: `php --version`

### "Class 'App\Models\User' not found"
- User model was created in Phase 5 stabilization
- Should exist at: `app/Models/User.php`

### Database Connection Error
- Verify DB settings in `.env`
- Create database: `CREATE DATABASE sheltra;`
- Run migrations: `php artisan migrate`

### CORS errors from frontend
- Check `FRONTEND_URL` in .env matches your frontend URL
- Frontend should be on http://localhost:3000

### Session not persisting
- Check `SESSION_DRIVER` is "file" or "database"
- Verify `storage/framework/sessions/` exists and is writable

## Integration with Frontend

Frontend (React on localhost:3000) will:
1. Make requests to `http://localhost:8000/api/*`
2. CORS headers are configured in `config/cors.php`
3. Cookies are sent automatically with requests
4. Session authentication is cookie-based

### Frontend Integration Points
- POST `/api/auth/register` - New user registration
- POST `/api/auth/login` - User login
- POST `/api/auth/logout` - User logout
- GET `/api/auth/me` - Get current user info
- GET `/api/refugee/profile` - Get refugee data
- And all other role-specific endpoints
